# HawkEye Backend Data Flow Documentation

This document traces the backend data flow from voice/text command to ADK routing to tool execution to BigQuery (when used) to WebSocket events.

Scope of observation:

- `app/main.py`
- `app/hawkeye_agent/agent.py`
- `app/hawkeye_agent/tools/analyst.py`
- `app/hawkeye_agent/tools/coordinator.py`
- `app/services/bigquery_service.py`
- `tests/test_live_agent.py`
- `tests/live_agent_test_results.md`
- direct BigQuery inspection run via `tests/inspect_bigquery.py`

Observed command tool-routing evidence comes from `tests/live_agent_test_results.md` final run.

---

## 0) Common Runtime Pipeline (applies to all commands)

1. Client sends command JSON over WebSocket:

```json
{"text":"<user command>"}
```

2. `app/main.py` (`websocket_endpoint` → `upstream_task`) parses text and pushes to ADK:

```python
live_request_queue.send_content(
    types.Content(
        role="user",
        parts=[types.Part.from_text(text=text_content)],
    )
)
```

3. ADK runner executes live turn:

```python
async for event in runner.run_live(...):
    await process_event(event, websocket, session_id)
```

4. `process_event()` emits:

- transcript events (`{"type":"transcript", ...}`)
- tool call events (`{"type":"tool_call", ...}`)
- tool result mapped events via `_map_tool_result_to_events()`
- turn boundary (`{"type":"turn_complete"}`)

5. Frontend receives raw WebSocket JSON events generated by `process_event()` and `_map_tool_result_to_events()`.

## 2) Command: "What happens if water rises 2 more meters"

### A) What the user says

`What happens if water rises 2 more meters`

### B) Which ADK agent receives it

- Root: `hawkeye_commander`
- Delegation target (observed): `analyst` sub-agent

### C) Which tool function is called

- `compute_cascade(flood_geojson, water_level_delta_m)` in `app/hawkeye_agent/tools/analyst.py`

### D) What that tool does internally

`compute_cascade()`:

1. Normalizes flood geometry via `_normalize_geojson_geometry(...)`
2. Calls `GroundsourceService.get_infrastructure_at_risk(normalized_geojson)`
3. Computes `buffer_m = int(water_level_delta_m * 500)` (2m => 1000m)
4. Calls `GroundsourceService.get_infrastructure_at_expanded_level(normalized_geojson, buffer_m)`
5. Calls `get_population_at_risk(...)` (EarthEngineService + density heuristic)
6. Builds cascade dict:
   - `first_order`
   - `second_order`
   - `third_order`
   - `fourth_order`
   - `summary`
   - `recommendation`

### E) BigQuery SQL executed (exact from `bigquery_service.py`)

Query used by `get_infrastructure_at_risk()`:

```sql
SELECT name, type, capacity, latitude, longitude,
       ST_ASGEOJSON(location) AS location_geojson,
       ROUND(ST_DISTANCE(location, ST_BOUNDARY(
         ST_GEOGFROMGEOJSON(@flood_geojson, make_valid => TRUE)
       )), 0) AS distance_to_flood_edge_m
FROM hawkeye.infrastructure
WHERE ST_INTERSECTS(location,
  ST_GEOGFROMGEOJSON(@flood_geojson, make_valid => TRUE))
ORDER BY type, distance_to_flood_edge_m
```

Query used by `get_infrastructure_at_expanded_level()`:

```sql
WITH expanded_flood AS (
  SELECT ST_BUFFER(
    ST_GEOGFROMGEOJSON(@flood_geojson, make_valid => TRUE),
    @buffer_meters
  ) AS expanded_geometry
),
currently_at_risk AS (
  SELECT id FROM hawkeye.infrastructure
  WHERE ST_INTERSECTS(location,
    ST_GEOGFROMGEOJSON(@flood_geojson, make_valid => TRUE))
)
SELECT i.name, i.type, i.capacity, i.latitude, i.longitude,
       ST_ASGEOJSON(i.location) AS location_geojson
FROM hawkeye.infrastructure i, expanded_flood ef
WHERE ST_INTERSECTS(i.location, ef.expanded_geometry)
  AND i.id NOT IN (SELECT id FROM currently_at_risk)
ORDER BY i.type
```

### F) What BigQuery returns (shape)

For `get_infrastructure_at_risk` SQL rows:

- `name`: STRING
- `type`: STRING
- `capacity`: STRING/NULL
- `latitude`: FLOAT64
- `longitude`: FLOAT64
- `location_geojson`: STRING (GeoJSON Point)
- `distance_to_flood_edge_m`: FLOAT64-like numeric

For `get_infrastructure_at_expanded_level` SQL rows:

- `name`: STRING
- `type`: STRING
- `capacity`: STRING/NULL
- `latitude`: FLOAT64
- `longitude`: FLOAT64
- `location_geojson`: STRING

### G) Tool transformation output

`GroundsourceService` groups rows by `type` and returns dicts keyed by:

- `hospitals`, `schools`, `shelters`, `power_stations`, `water_treatment`

`compute_cascade()` then transforms those grouped results into:

```json
{
  "first_order": {"population_at_risk": <int>, "water_level_delta_m": <float>, ...},
  "second_order": {"hospitals_at_risk": <int>, "hospital_names": [...], "newly_isolated_hospitals": [...]},
  "third_order": {"power_stations_at_risk": <int>, "estimated_residents_without_power": <int>, ...},
  "fourth_order": {"children_under_5": <int>, "elderly_over_65": <int>, "hospital_patients_needing_evac": <int>},
  "summary": "<text>",
  "recommendation": "<text>"
}
```

### H) `_map_tool_result_to_events()` mapping

For `tool_name == "compute_cascade"`, `main.py` emits:

1. `status_update` with:
   - `population`
   - `waterLevel`
   - `hospitals_at_risk`
   - `power_stations_at_risk`
   - `children_under_5`
   - `elderly_over_65`
2. `incident_log_entry` with cascade summary and recommendation
3. `map_update` marker layer `cascade_infrastructure` if `newly_isolated_hospitals` exists

Representative payloads:

```json
}
```


## 3) Command: "Route evacuation from Kampung Melayu to nearest shelter"

### A) What the user says

`Route evacuation from Kampung Melayu to nearest shelter`

### B) Which ADK agent receives it

- Root: `hawkeye_commander`
- Delegation target (observed): `coordinator` sub-agent
- Additional safety tool call (observed): `evaluate_route_safety` (Analyst tool exposed in coordinator toolset)

Observed final run tool sequence:

1. `generate_evacuation_route`
2. `evaluate_route_safety`
3. `log_incident`

### C) Which tool function is called

- `generate_evacuation_route(origin_name, destination_name, avoid_flood=True)` in `app/hawkeye_agent/tools/coordinator.py`
- `evaluate_route_safety(route_geojson, flood_geojson)` in `app/hawkeye_agent/tools/analyst.py`
- `log_incident(...)` in `app/hawkeye_agent/tools/coordinator.py`

### D) What these tools do internally

`generate_evacuation_route`:

1. `MapsService.geocode(origin_name)`
2. `MapsService.geocode(destination_name)`
3. `MapsService.get_evacuation_route(origin_latlng, destination_latlng)` (Directions API)
4. Returns route GeoJSON + distance/duration and metadata

`evaluate_route_safety`:

1. Parse route and flood GeoJSON via `_parse_geojson_payload`
2. Spatial intersection with Shapely
3. Compute `intersection_pct` and `safety_rating` (`SAFE|CAUTION|UNSAFE`)
4. Build danger zone markers
5. Optionally call `_suggest_alternative_route` (Maps API) when unsafe

`log_incident`:

- Writes structured incident to Firestore via `IncidentService.log_event(...)`

### E) BigQuery SQL executed

None in this route path.

### F) What BigQuery returns

Not applicable (no BigQuery query in this path).

### G) Tool transformation output

`generate_evacuation_route` output:

```json
{
  "route_geojson": {"type":"Feature","geometry":{"type":"LineString","coordinates":[...]},"properties":{...}},
  "distance_m": <int>,
  "duration_minutes": <float>,
  "origin": {"lat":...,"lng":...,"formatted_address":"..."},
  "destination": {"lat":...,"lng":...,"formatted_address":"..."},
  "safety_rating": "PENDING_ANALYSIS",
  "avoid_flood": true
}
```

`evaluate_route_safety` output includes:

- `is_safe`
- `safety_rating`
- `intersection_pct`
- `danger_zones`
- `recommendation`
- optional `alternative_route`

### H) `_map_tool_result_to_events()` mapping

- `generate_evacuation_route` -> `map_update` overlay layer `evacuation_route`
- `evaluate_route_safety` -> if unsafe/caution with danger points, `map_update` markers layer `route_danger_zones`
- `log_incident` -> `incident_log_entry`

## Real vs Mocked Data Sources

### Groundsource flood history

- Status: **REAL**
- Evidence: `GroundsourceService` runs live BigQuery SQL against `hawkeye.groundsource_jakarta`.
- To keep fully real: maintain BigQuery dataset freshness and service account access.

### Jakarta infrastructure (hospitals, schools, shelters, power)

- Status: **REAL (backend runtime) + PRE-COMPUTED fallback (frontend static export)**
- Evidence:
  - Backend: live BigQuery query to `hawkeye.infrastructure`
  - Frontend: `frontend/src/data/infrastructureData.js` is a static export snapshot
- To make fully real end-to-end: ensure frontend always consumes backend `/api/infrastructure` and remove reliance on static snapshot fallbacks.

### Flood extent polygon

- Status: **PRE-COMPUTED (currently synthetic placeholder)**
- Evidence:
  - `data/geojson/flood_extent.geojson` has `"source": "synthetic-placeholder"`
  - `EarthEngineService` reads this file directly
- To make fully real: run `data/compute_flood_extent.py` with authenticated Earth Engine and update file/provenance on a schedule.

### Triage zones

- Status: **PRE-COMPUTED**
- Evidence: `data/geojson/triage_zones.geojson` static FeatureCollection with manual zone metadata.
- To make fully real: generate zones dynamically from live flood extent + elevation + population + infrastructure reachability model.

### Population density

- Status: **MOCKED**
- Evidence:
  - `frontend/src/components/strategic/DataLayerPanel.jsx` generates synthetic Gaussian hotspot grid (`generatePopulationGrid()`).
  - Also mock data defaults in frontend.
- To make fully real: serve raster/grid from real demographic datasets (BPS census blocks, WorldPop, or Meta HRSL) through backend APIs.

### Earth Engine satellite analysis metrics

- Status: **MIXED PRE-COMPUTED + MOCKED**
- Evidence:
  - `EarthEngineService.get_flood_extent_geojson()` reads local file.
  - `get_flood_growth_rate()` returns hardcoded `12.0` with `"source":"pre-computed"`.
  - `get_flood_area_sqkm()` is computed geometrically from the local polygon.
- To make fully real: compute growth rate from time-series Earth Engine imagery and expose run timestamp/versioned analysis artifacts.

### Evacuation routes

- Status: **REAL**
- Evidence: `MapsService.get_evacuation_route()` calls Google Maps Directions API (`client.directions(...)`) live at runtime.
- To keep fully real: maintain Maps API key + enabled Directions API.

### Geocoding

- Status: **REAL**
- Evidence: `MapsService.geocode()` calls Google Maps Geocoding API (`client.geocode(...)`).
- To keep fully real: maintain Maps API key + enabled Geocoding API.

### Weather data

- Status: **MOCKED / NOT INTEGRATED**
- Evidence:
  - No dedicated weather service module in `app/services/`
  - No structured weather API pipeline in backend tools
  - Only generic mention in search-grounding text
- To make fully real: add a weather service/tool (BMKG/OpenWeather/NOAA), integrate into agent routing, and emit structured weather events.

### Nano Banana 2 projections

- Status: **REAL (live model call)**
- Evidence: `predictor.generate_risk_projection()` calls Gemini image generation (`genai.Client(...).models.generate_content(...)`) with image+text input.
- Caveat: output is AI-generated approximation, not physics simulation.
- To strengthen realism: pair generated visuals with hydrology model outputs and uncertainty metadata.
